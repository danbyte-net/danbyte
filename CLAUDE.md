# Danbyte Agent Guide

This file is the working contract for coding agents in this repository. Read it
before making changes, then read the topical docs and nearby code for the task.

## What Danbyte Is

Danbyte is a customizable IPAM/DCIM and network-operations platform. It models
IP space, tenants, sites, devices, racks, interfaces, cabling, circuits, power,
floor plans, topology, monitoring, compliance, automation, and audit history.

The product is preparing for public preview and already manages real
infrastructure. Preserve existing data and compatibility; do not treat the
database, migrations, APIs, or UI as disposable scaffolding.

## Architecture

### Backend

- Python 3.13, Django 5.2 LTS, Django REST Framework, PostgreSQL, Redis, RQ.
- `danbyte/`: settings, top-level URLs, WSGI, and ASGI entry points.
- `api/`: the main IPAM/DCIM/connectivity model and REST surface.
- `core/`: organization/tenant settings, shared mixins, tags, and bookmarks.
- `auth_api/`: login, profiles, RBAC, API tokens, and LDAP.
- `customization/`: custom-field definitions, validation, and object registry.
- `monitoring/`: checks, scheduling, state, alerts, engines, and SNMP.
- `integrations/`: webhooks, automation/deploy runs, drift, and NetBox import.
- `audit/`, `compliance/`, `search/`, and `jobs/`: their named concerns.
- `danbyte_checks/`: Django-free protocol checker package shared with Outposts.
  Put shared check behavior here instead of forking core and agent logic.

Django owns `/api/`, `/admin/`, `/django-rq/`, static/media, and WebSockets.
HTTP production traffic uses gunicorn; WebSockets use a separate daphne
process. Do not enable daphne/channels as the normal Django `runserver` backend;
the separation is deliberate and documented in `danbyte/settings.py`.

### Frontend

- React 19, TypeScript, TanStack Start/Router/Query/Table, Vite, Tailwind 4,
  shadcn/Radix, and Lucide.
- `frontend/src/routes/`: file-based routes.
- `frontend/src/components/`: shared and domain components.
- `frontend/src/lib/api.ts`: primary typed API boundary.
- `frontend/src/styles.css`: active design tokens and global styling.
- Vite on `:3000` is the user-facing dev server and proxies Django paths to
  `:8000`. Authentication is Django session + CSRF, or scoped API tokens. There
  is no JWT flow.

Production nginx routes the SPA, API, WebSocket, admin, static, media, and docs
services behind one origin.

## Non-Negotiable Product Rules

### Customization First, With Deliberate Seeds

"Zero pre-filled data" is a direction, not an absolute ban. Danbyte should not
impose arbitrary network inventory or an arbitrary operating model, but it may
seed data needed for the product to work, remain safe, or provide a sane
first-run experience.

Classify data before adding it:

- Required system/bootstrap data, such as access roles, structural catalogs,
  protocol constants, or records required by application invariants, may be
  seeded.
- Useful starting catalogs may be seeded when there is a clear product reason,
  especially when users can edit, replace, or remove them safely.
- Example networks, devices, prefixes, VLANs, tags, and other illustrative
  content belong in opt-in demo seeders, never normal production bootstrap.

Seeds must be minimal, deterministic, idempotent, tenant/deployment scoped as
appropriate, and safe on both fresh installs and upgrades. Prefer editable
catalog records over hard-coded choices when customization matters. Follow the
existing bootstrap or data-migration pattern, add tests, and document why a new
seed is required. Do not reject a necessary seed solely by quoting the
zero-data slogan.

### Tenant, Site, And RBAC Safety

- Treat tenant isolation as a hard security boundary.
- Scope every domain queryset to the active/authorized tenant before filtering
  by user-controlled IDs.
- Never trust a UUID, foreign key, relation, bulk payload, import row, or picker
  result to belong to the current tenant/site.
- Enforce access on the server. Hidden buttons are only a UI convenience.
- Preserve site-scoped write rules, catalog locality, and enhanced site
  separation behavior.
- New DRF endpoints remain default-closed. Use `AllowAny` only for an explicitly
  public flow with focused tests.
- Validate cross-tenant relationships on create, update, bulk operations,
  imports, clone/sync paths, and background jobs.

Security-sensitive settings fail closed. Do not weaken secret-key, credential
encryption, TLS/cookie, CSRF, host, or SSRF protections for convenience. Do not
return write-only secrets from serializers or bypass the outbound-request
guards.

### Data And Schema

- PostgreSQL is required; do not add SQLite-specific fallbacks.
- Domain models use UUID primary keys.
- Preserve load-bearing constraints such as nullable VRF semantics and
  `nulls_distinct=False` uniqueness.
- Add forward migrations for schema changes. Never delete or rewrite committed
  migrations, wipe a database, or reseed user data unless the user explicitly
  requests a disposable-environment reset.
- Keep demo data opt-in and idempotent. Production migrations may create
  required system data, but never illustrative/demo inventory.

## How To Work

Start with the graph and ownership boundaries, then inspect only the symbols
needed for the task. When Ix Memory is available, use `ix-understand` for broad
orientation, `ix-investigate` for a symbol, `ix-impact` before non-trivial
edits, and `ix-debug` for a symptom. Separate graph-backed facts from
inferences.

Before editing:

1. Read `git status` and preserve all unrelated user changes.
2. Read the relevant feature doc, route/component or viewset/serializer, tests,
   and neighboring established pattern.
3. Check the blast radius across API contracts, RBAC/tenant scoping, background
   work, frontend consumers, migrations, and docs.

Keep changes narrow. Reuse existing helpers and primitives. Do not mix unrelated
refactors, formatting churn, dependency upgrades, or generated artifacts into a
feature/fix.

## Backend Conventions

- Follow existing serializer/viewset/service boundaries in the affected app.
- Keep validation close to the contract and return actionable field errors.
- Use shared RBAC, tenant/site scoping, audit, secret, and SSRF helpers instead
  of duplicating partial checks.
- Preserve audit/journal behavior for user-visible mutations, including bulk
  and async paths.
- Use RQ for established background workflows. Jobs must re-check scope and
  permissions where appropriate; do not assume enqueue-time IDs stay valid.
- Never use a shell for monitoring exec checks. Keep the existing allow-listed,
  no-shell plugin execution model and its disabled-by-default gate.
- Add focused regression tests for fixes and broader tests when shared behavior,
  permissions, tenancy, migrations, or public APIs change.

Python style is configured in `pyproject.toml`: Ruff, Python 3.13, 100-column
format target, with migrations excluded.

## Frontend Conventions

Use the active shared system; do not hand-roll another version of existing UI:

- Lists: `ListPageShell`, `DataTable`, shared filters, and entity column
  factories under `components/columns/`.
- Details: `DetailShell`, Overview first, then related tabs, with Journal and
  History last. Use `KvCard` for attribute groups.
- Object selection: `ObjectPicker` or an existing domain picker preset.
- State and errors: `QueryError`, `apiErrorToast`, and `EmptyState`.
- Colored catalog objects: `ColorBadge`, `StatusBadge`, `RoleChip`, or the
  existing domain badge. Derive color from object data, never its name.
- Dates/times: `TimeCell` or the shared time helpers.
- Tabs: the shared segmented-tabs component.

One entity should have one column factory, reused by list and embedded tables.
Do not copy a shell, inline a second `ColumnDef[]`, add raw `<table>` markup, or
invent a different loading/error/empty treatment when a shared primitive fits.

Loading rules:

- Page/section fetch: muted `Loading...` text or the table's built-in row.
- Pending labeled button: keep its dimensions and change the verb to
  `Saving...`, `Deleting...`, etc.
- Spinner: icon-only buttons only.
- Skeletons: existing sidebar and topology-canvas uses only.

Keep the UI restrained and operator-focused. Use semantic CSS variables and
the established zinc-based palette, color only for meaning, `rounded-md` or
`rounded-lg`, and shadows only for real overlays. Use Lucide icons from the
existing library. Avoid gradients, decorative shadows, colored dot-plus-name
status treatments, hard-coded status-name palettes, and one-off SVG icons.

The full design system — tokens, palette, type scale, and component patterns —
lives in `docs/design/visual-language.md` and
`docs/architecture/shadcn-tokens.md`. Read them before adding UI.

This is a shadcn project (`frontend/components.json`). Reuse local components
and registry conventions; do not rerun `shadcn init` or overwrite customized
primitives.

## Documentation

A user-visible or operational change is incomplete without docs in the same
change. Update the relevant page when changing a model field, API/view
behavior, form/payload, UI control, route, setting/env var, Make target, or
service. Match the surrounding docs for location, frontmatter, and style.

Use `docs/` as source. Do not edit generated `/site/`. Update older architecture
pages when a change would otherwise deepen the documented/current-state gap.

## Commands

Use the repository virtualenv and the package manager used by CI.

```bash
# Infrastructure and backend
make infra-up
make migrate
make bootstrap
make backend-up
make workers-up

# Backend verification
make check
make test
make coverage
.venv/bin/python manage.py test <app.tests_or_test_module>
.venv/bin/ruff check <changed paths>

# Frontend (run separately; `make up` does not start it)
make frontend-install
make frontend-dev
cd frontend && npm test
cd frontend && npm run build
cd frontend && npm run typecheck
cd frontend && npm run lint
cd frontend && npm run check

# Docs
make docs-up
make docs-build
```

Run frontend build before typecheck because the build regenerates the TanStack
route tree. CI uses `npm ci` and `frontend/package-lock.json`; treat npm as the
effective package manager. The simultaneous pnpm lock/workspace is known debt,
so do not churn both lockfiles incidentally.

CI does not necessarily run the full Django test suite: it is gated by the
`RUN_TESTS` repository variable. Run relevant backend tests locally. Frontend
build and typecheck, plus backend migration application and
`makemigrations --check`, are current CI gates; lint/format checks are still
advisory but should not gain new violations.
